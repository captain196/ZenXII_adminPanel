2. New Parent Nodes (3 Nodes)

2.1   System/ — Platform-Level Data

Stores everything that belongs to the platform as a whole, not to any individual school. This is the biggest addition.

Sub-pathWhat It StoresUsed By
System/Schools/{SCH_XXXXXX}/ Canonical school registry — profile, subscription, plan, status, stats cache SA panel, login, health checks, school config
System/Plans/{PLAN_ID}/ Subscription plan definitions — name, price, modules, billing cycle, grace days SA Plans management, onboarding
System/Payments/{PAY_ID}/ Payment transaction records — amount, gateway, status, invoice SA payment processing
System/Backups/{uid}/{id} Backup metadata — filename, type, size, status, encryption key SA backup system, cron job
System/BackupSchedule Automated backup config — frequency, retention, last run Backup cron job
System/Logs/Activity/ SA action audit trail (date-partitioned) SA monitoring dashboard
System/Logs/Payroll/ Payroll computation audit trail HR module
System/API_Keys/{hash} Device API key → school lookup (biometric/RFID) Attendance module

2.2   RateLimit/ — Brute-Force Protection

Sub-pathPurposeThreshold
RateLimit/Login/{ip} Tracks failed school admin login attempts per IP 20 fails / 15 min
RateLimit/SA/{ip} Tracks failed SA login attempts per IP 10 fails / 30 min

Stores {fails: count, windowStart: timestamp}. Cleared on successful login. Purely security infrastructure.

2.3   Indexes/ — Fast School Lookups

Indexes/School_names/{name} name → SCH_XXXXXX Onboarding: uniqueness check by school name

6. Key Takeaway

Schools/ and Users/ remain the core of your database — nothing changed there.

The 3 new nodes add platform infrastructure (subscriptions, security, auditing, backups) that didn't exist before:

  • System/ — centralizes platform-level data that doesn't belong to any single school
  • Indexes/ — enables O(1) school resolution instead of scanning
  • RateLimit/ — adds brute-force login protection

School_ids/ is the only old node with a direct replacement (Indexes/School_codes/), and both are still read for backward compatibility.

Subject_master_list is unused and can be safely deleted from Firebase.
