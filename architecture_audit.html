Admin Portal Architecture Audit

⚙

Architecture Type: Modular Monolith + External Auth Microservice

┌───────────────────────────────────────────────────────────────┐
│                      ADMIN WEB PORTAL                         │
│  CodeIgniter 3 (PHP) — Modular Monolith                      │
│                                                               │
│  MY_Controller (base)                                       │
│    ├── Auth check (session)                                   │
│    ├── Firebase init ($this->firebase)                        │
│    ├── School context ($this->school_name,                    │
│    │    $this->school_code, $this->parent_db_key)              │
│    ├── Session context ($this->session_year)                  │
│    ├── RBAC enforcement (_require_role)                       │
│    └── CSRF on all POST                                       │
│                                                               │
│  jQuery AJAX → PHP Controller → Firebase RTDB                 │
│  (no REST standard — mixed GET/POST, HTML + JSON)            │
└──────────────┬────────────────────────────────────────────────┘
               │ /internal/* (X-Internal-Key)
┌──────────────▼────────────────────────────────────────────────┐
│               NODE.JS AUTH API (Render.com)                  │
│  Express 5 — JWT auth, device binding, OTP                    │
│  MongoDB (graderIQ) ←→ Firebase RTDB (dual-write)            │
└───────────────────────────────────────────────────────────────┘

Controller → DB Flow

Browser → jQuery $.ajax() → PHP Controller method → $this->firebase->get/set/update/delete(path) → Firebase Realtime Database → json_success() / json_error() back to browser
Controllers serve both HTML pages (GET) and JSON endpoints (POST AJAX). No versioning, no consistent response envelope outside json_success/json_error.

Real Data Flows

1

Flow 1: Student Admission → Full Lifecycle

Step 1 — Admin fills admission form

POST sis/save_admission → Sis::save_admission()
Generates STU_ID via Auth_client::sync_student()
→ POST /internal/sync-student (Node.js)
→ MongoDB: users.insert({userId:"STU0042", password:bcrypt, schoolId:"10005"...})
→ Firebase: Users/Parents/{parentDbKey}/STU0042/{full profile}

Step 2 — Admin enrolls student into class

POST sis/enroll_student
→ Firebase SET: Schools/{school}/{session}/Class 9th/Section A/Students/STU0042 = true
→ Firebase UPDATE: Users/Parents/{key}/STU0042/Class = "9th", Section = "A"
→ Firebase UPDATE: SIS/Students_Index/STU0042/status = "Active"

Step 3 — Student now visible in all downstream modules

Attendance (class roster) • Fees (student lookup) • Exams (marks entry) • Communication (recipients) • Transport/Hostel (assignments)

Step 4 — MongoDB record enables mobile

Mobile app login: STU0042 + password → JWT tokens
Device binding: first device auto-bound
Parent sees child's data immediately

2

Flow 2: Fee Creation → Payment → Accounting

Step 1 — Admin defines fee structure

POST fee_management/save_fee_structure
→ Firebase SET: Accounts/Fees/Fees Structure/Monthly/Tuition = 5000

Step 2 — Admin maps fees to classes

POST fee_management/save_class_fees
→ Firebase SET: Accounts/Fees/Classes Fees/Class 9th/Section A/ = {Tuition:5000, Transport:1500}

Step 3 — Fee details fetched

POST fees/fetch_fee_details (userId, months[])
Reads class fees + already paid months + discounts
Returns: {monthly:6500, total:13000, discount:500, net:12500}

Step 4 — Payment submitted

POST fees/submit_fees
→ Creates voucher in Accounts/Vouchers/{receipt}
→ Updates Users/Parents/{key}/{STU}/Fees Record/ + Month Fee/
→ Deletes from Accounts/Pending_fees/{STU}/

Step 5 — Accounting journal auto-created

Operations_accounting::create_fee_journal()
→ Dr: 1010 (Cash) = ₹12,500 • Cr: 4010 (Tuition Income) = ₹10,000 + 4020 (Transport) = ₹2,500
→ Closing balances updated

Step 6 — Communication trigger

fire_event("fee_received", {student_id, amount:12500})
→ Queue → Process → Push notification to parent app

3

Flow 3: Teacher Marks Entry → Parent Visibility

Step 1 — Exam + template created

POST exam/create_exam → Exams/{EX001} = {name:"Mid-Term", status:"Published"}
POST result/save_template → Components: Theory(80) + Practical(20)

Step 2 — Teacher enters marks

POST result/save_marks
→ Results/Marks/EX001/Class_9th/Section_A/English/STU0042 = {Theory:65, Practical:18, Total:83}

Step 3 — Admin computes results

POST result/compute_results
→ Computes: percentage, grade, pass/fail, rank (competition: 1,1,3)
→ Results/Computed/{exam}/{class}/{section}/{STU} = {Pct:82.9, Grade:"A", Rank:3}

Step 4 — Parent sees results in real-time

Firebase listener on Results/Computed/ fires automatically
Communication trigger → push notification sent

4

Flow 4: Configuration Propagation

Admin adds subject "Computer Science"

POST school_config/save_subject → Subject_list/9/CS = "Computer Science"

Propagates to: Academic (subject assignments), Exam (subject list), Result (template designer), Timetable (period assignment), Teacher App (schedule)

Admin creates new session "2026-27"

POST school_config/add_session → Config/ActiveSession = "2026-27"

Impact: ALL controllers use new session_year. Fresh class structure needed. Old data intact. Students need promotion.

│   (PHP)      │    │  (Mobile)   │    │  (Mobile)   │

Conflict Handling

  • Attendance: Firebase transaction() — atomic read-modify-write with auto-retry
  • Marks: Last-write-wins (per-teacher-per-subject, no real concurrency)
  • Fees: Only admin writes vouchers — parent app read-only
  • Messages: Append-only (push) — no conflict possible

Offline Handling

  • Parent App: Firebase disk persistence ON — reads from cache, shows stale indicator
  • Teacher App: Firebase disk persistence ON — writes queued locally, auto-synced on reconnect
  • Why it works: Firebase RTDB has built-in offline support — no custom sync engine needed

Current Strengths

AspectStatusDetail
Multi-tenancyGoodComplete isolation via Schools/{school_id}/ prefix
Session isolationGood{session_year}/ prevents cross-year contamination

Problems Found

ProblemSeverityDetail
No Firebase Security RulesCriticalAdmin SDK bypasses rules. Mobile apps need client-side Firebase with security rules. Without rules, any user can read/write any school's data.
Attendance as stringMedium"PPAPLHV..." — modifying day 15 requires read-modify-write of entire string. Concurrent overwrite risk with teacher app.
Fee paths scatteredMediumFee data in 5+ locations. Mobile app must read multiple paths to assemble complete picture.
No paginationMediumCommunication/Notices/ could grow large. Need .limitToLast(N) + cursor.
Dual-write riskMediumsync_student() writes to MongoDB + Firebase. Partial failure = inconsistent state.

Required: Firebase Security Rules

MUST implement before mobile launch. Currently all data is unprotected on the client side.
Firebase Security Rules
{ "rules": { "Users": { "Parents": { "$schoolKey": { "$studentId": { ".read": "auth != null && auth.token.parentDbKey == $schoolKey && auth.token.userId == $studentId", ".write": false } } } }, "Schools": { "$schoolId": { ".read": "auth != null && (auth.token.schoolCode == $schoolId || auth.token.schoolId == $schoolId)", "$session": { "Results": { "Marks": { "$examId": { "$class": { "$section": { "$subject": { ".write": "auth != null && auth.token.role == 'teacher'" }}}} } } } } } } }

Required: Firebase Custom Token Generation

Add to Node.js Auth API — mobile-app-login response
// After successful JWT login, also generate Firebase custom token: const firebaseToken = await admin.auth().createCustomToken(userId, { schoolCode: user.schoolCode, schoolId: user.schoolId, parentDbKey: user.parentDbKey, role: user.role, userId: user.userId }); // Mobile app then calls: firebase.auth().signInWithCustomToken(firebaseToken);

Recommended Architecture — DO NOT OVER-ENGINEER

  • Mobile apps read Firebase RTDB directly (with security rules + custom auth tokens)
  • Mobile apps write Firebase RTDB directly (teacher: attendance, marks — with security rules)
  • Node.js Auth API handles: login + JWT + device binding + Firebase token + FCM + 2-3 aggregate endpoints
  • PHP Admin Panel continues unchanged — writes to same Firebase paths, mobile sees changes in real-time
  • DON'T build a new REST API layer on top of PHP
  • DON'T create a separate "mobile backend"
  • DON'T migrate to Firestore or a new database
This works because Firebase IS the sync layer. All three systems read/write the same database. Firebase real-time listeners handle the sync automatically.
